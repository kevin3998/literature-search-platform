# Phase 1 PostgreSQL User Boundary Design

Date: 2026-07-09

## Purpose

Phase 1 turns the platform business database from a local SQLite development store into a product-ready PostgreSQL foundation. It introduces a personal-account data boundary, stable internal UUID identities, per-user settings and encrypted credentials, a PostgreSQL-backed job queue, and deployment-ready operational checks.

This phase does not implement the final login UI, team workspaces, organization membership, or migration of the literature corpus/index database.

## Scope

In scope:

- Replace the platform business runtime database with PostgreSQL.
- Use Alembic and SQLAlchemy Core for schema migrations, connections, and transactions.
- Create `users` and `user_identities` as the authentication foundation.
- Store all platform business entity IDs as PostgreSQL `uuid` values generated by the application.
- Scope all business data to the current internal `user_id`.
- Make settings, model profiles, and API keys per-user.
- Encrypt user API keys before storing them in PostgreSQL.
- Move background task execution to a PostgreSQL-backed queue with an independent worker process.
- Add a session bundle endpoint to replace multi-request session opening.
- Add `/api/me`, user-scoped frontend local storage, and a development-only user switcher.
- Add readiness checks, backup script, contract tests, and deployment documentation.

Out of scope:

- Final login page, account menu, password login, JWT/OIDC implementation, or SSO.
- Team, organization, shared workspace, role membership, or collaboration features.
- Migration of existing SQLite development data into PostgreSQL.
- Migration of the literature corpus, `research_index.sqlite`, PDFs, embeddings, or indexing pipeline.
- Full user data export or hard-delete workflows.
- Multi-instance backend deployment, Redis, Celery, object storage, or distributed SSE.

## Architecture Decisions

### Database

PostgreSQL becomes the only supported runtime database for platform business state. SQLite platform-business code may remain temporarily as legacy code but must not be used as a runtime fallback.

The application uses a dedicated PostgreSQL schema:

```text
literature_agent
```

Connections set:

```sql
search_path = literature_agent, public
```

Tests create independent temporary schemas and point `search_path` to the test schema.

The initial Alembic migration is a fresh PostgreSQL schema, not a SQLite-to-PostgreSQL incremental migration.

Schema conventions:

- `snake_case` table and column names.
- `uuid` primary keys and foreign keys.
- `jsonb` for JSON payload fields.
- `timestamptz` for database time columns.
- API responses continue returning Unix seconds for timestamp compatibility.
- UUIDs are returned as lowercase standard strings with hyphens.

### Identity

The platform uses internal stable UUID users and external identity mappings.

```text
users
user_identities
```

Business tables store only internal `users.user_id`. External identifiers such as `local_user`, `alice`, email addresses, or OIDC subjects are stored only in `user_identities`.

Development mode:

```text
AUTH_MODE=dev-header
```

In this mode, `X-User-Id` is treated as a development identity subject. Missing `X-User-Id` maps to `local_user`. The app automatically creates the internal user and identity mapping if needed.

Production mode must reject insecure development authentication. If:

```text
APP_ENV=production
AUTH_MODE=dev-header
```

the backend must refuse to start.

Future authentication modes such as `jwt`, `oidc`, or `external` are reserved for later phases.

### User Model

Phase 1 uses a personal-account model:

```text
user owns sessions, jobs, settings, model profiles, workflows, extraction tasks, and artifacts metadata
```

There is no team, workspace membership, or organization model in Phase 1.

`users.display_name` is the user-facing label shown in the UI. Internal UUIDs are not shown as user labels.

The UI may expose current user information through:

```text
GET /api/me
```

Example:

```json
{
  "user_id": "8b8b9bb8-4e1a-4e23-9b9e-1f0d0f4f3abc",
  "display_name": "local_user",
  "auth_mode": "dev-header"
}
```

### Data Ownership

All core business tables include `user_id` either directly or through an owned parent entity.

Per-user resources include:

- settings
- model profiles
- encrypted secrets
- sessions
- messages and turns
- attachments
- search results and evidence items
- research state and curation state
- jobs and job events
- artifacts metadata
- workflow runs and workflow steps
- structured extraction tasks and child records

Settings are unique by:

```text
(user_id, scope, key)
```

Model profiles are user-scoped. Profile names are unique only within a user. Each user may have at most one active model profile, enforced by a partial unique index.

Cross-user access returns `404 Not Found` externally, not `403`, so resource existence is not leaked. The denial is recorded internally in audit events.

### API IDs

All platform business entity IDs become UUIDs:

- users
- sessions
- messages
- turns
- jobs
- workflows
- artifacts
- structured extraction tasks
- model profiles
- attachments
- related child entities

The application generates UUIDs before insert. PostgreSQL columns are still typed as `uuid`.

API responses keep domain-specific ID fields and also expose a generic `id` where appropriate:

```json
{
  "session_id": "8b8b9bb8-4e1a-4e23-9b9e-1f0d0f4f3abc",
  "id": "8b8b9bb8-4e1a-4e23-9b9e-1f0d0f4f3abc"
}
```

Old prefixed IDs such as `s_xxx`, `job_xxx`, and `wf_xxx` are not supported in PostgreSQL mode. The frontend clears stale non-UUID local storage entries during startup.

### Foreign Keys And Deletion

Users are not physically deleted in Phase 1. User-level foreign keys should not cascade-delete all business data.

Business object child tables may cascade from their parent:

- session hard delete cascades to messages, turns, attachments, research state, and related session child data.
- job hard delete cascades to job events.
- workflow hard delete cascades to workflow steps.
- structured extraction task hard delete cascades to its task-local child records.

The normal product behavior remains soft deletion through `deleted_at`, `archived`, or status fields.

`audit_events` keeps weak resource references and does not cascade with business resource deletion.

### Secrets

User API keys and model credentials are encrypted before they are stored in PostgreSQL.

The encryption mechanism is Fernet from the Python `cryptography` package.

Secret records include key/version metadata so future key rotation can be added without schema changes:

```text
encryption_version
key_id
```

Development:

- If `DATA_ENCRYPTION_KEY` is missing, the app creates and reuses `.runtime/secret.key`.

Production:

- `DATA_ENCRYPTION_KEY` is required.
- Missing production encryption configuration blocks startup.

The UI and API do not reveal plaintext API keys. Users can save, overwrite, delete, test, and view masked previews only.

### Files And Artifacts

PostgreSQL stores artifact metadata only. Artifact bodies, exports, structured extraction workspaces, and similar files remain in the filesystem.

Platform-owned user files use:

```text
PLATFORM_DATA_DIR
```

Development default:

```text
.runtime/platform_data
```

Production recommendation:

```text
/var/lib/literature-agent-platform
```

User files are rooted at:

```text
PLATFORM_DATA_DIR/users/<user_uuid>/...
```

The database stores paths relative to `PLATFORM_DATA_DIR`, not absolute host paths.

The literature corpus and index remain separate:

```text
LITERATURE_DATA_DIR
research_index.sqlite
PDF/source files
```

They are not migrated in Phase 1.

### Permissions

Routes use the current authenticated `UserContext`; clients do not pass target `user_id` values to access resources.

Introduce shared ownership helpers such as:

```text
require_session
require_workflow
require_extraction_task
require_job
```

These helpers:

- Query using the current internal `user_id`.
- Return or raise `404`.
- Record `audit_events` for denied cross-user access where detectable.

### Audit Events

Add a minimal `audit_events` table.

It records key security and ownership events:

- development identity auto-created
- settings updated
- model profile created, updated, deleted, or activated
- secret stored, deleted, or tested
- denied cross-user access
- authentication mode safety failures or readiness warnings

No audit UI is required in Phase 1.

### Session Bundle

Add:

```text
GET /api/sessions/{session_id}/bundle
```

This endpoint returns the data needed to open a session:

```json
{
  "session": {},
  "messages": [],
  "context": {},
  "artifacts": [],
  "jobs": [],
  "research_state": {},
  "attachments": []
}
```

The frontend uses this endpoint instead of opening a session with several concurrent requests. Existing granular endpoints may remain for focused refreshes and backwards-compatible internal use.

### Job Queue And Worker

Introduce a PostgreSQL-backed queue using the `jobs` table as the scheduling source.

The web process enqueues jobs and reads job state. A separate worker process claims and executes jobs:

```text
web:    uvicorn main:app
worker: python -m worker
```

Job claiming uses:

```sql
FOR UPDATE SKIP LOCKED
```

The worker uses a centralized `JobHandlerRegistry`:

```text
job_type -> handler
```

First-phase job types include:

- corpus maintenance jobs
- workflow run and workflow step execution
- structured extraction run
- structured extraction evidence packet build
- structured extraction multimodal review
- literature deep research / existing job runner style tasks

Not included:

- ordinary synchronous API reads
- literature index read-only queries
- normal chat streaming

Normal chat streaming remains in the web request because it is latency-sensitive and streams directly to the client.

Worker behavior:

- independent process
- heartbeat written to PostgreSQL
- configurable `WORKER_CONCURRENCY`, default `1`
- queued jobs can be cancelled
- running jobs use cooperative cancellation
- transient errors have limited automatic retry
- permanent errors fail immediately
- SSE remains the realtime progress mechanism and reads from `job_events`

### Frontend

Frontend startup sequence changes to:

```text
GET /api/me
load current user
read user-scoped localStorage
load modules, sessions, settings, and current surface
```

All user-related local storage is scoped by internal `user_id`.

The existing development `X-User-Id` API client support remains only for `AUTH_MODE=dev-header`.

Developer mode includes a user switcher:

- It sets the development identity subject.
- It reloads the page.
- It is explicitly temporary and must later be replaced by a real account menu and authentication flow.

The UI displays `users.display_name`.

### Readiness And Startup Safety

Startup must block dangerous configuration:

- production with `AUTH_MODE=dev-header`
- production without `DATA_ENCRYPTION_KEY`
- missing or non-PostgreSQL `DATABASE_URL`
- production database schema not at Alembic head

Readiness reports unhealthy state with `503` for runtime dependencies:

- PostgreSQL unavailable
- Alembic version not current
- `PLATFORM_DATA_DIR` not writable
- worker heartbeat missing/stale
- development default user missing

`/api/readiness` includes checks for:

- PostgreSQL connection
- migration head
- authentication mode safety
- encryption key configuration
- current-user session read contract
- user data directory writability
- worker heartbeat

### Deployment

Local development:

```bash
docker compose up -d postgres
./dev.sh
```

`docker-compose.yml` includes PostgreSQL only.

`dev.sh` handles:

- `DATABASE_URL` checks
- development auth defaults
- development secret key generation
- `alembic upgrade head`
- FastAPI web startup
- worker startup
- Vite frontend startup

Production deployment uses:

- PostgreSQL service
- Python virtual environment
- systemd web service
- systemd worker service
- Vite production build
- Nginx or Caddy for static frontend and reverse proxy

Production does not auto-run migrations during service startup. Deployment must run migrations explicitly before starting or restarting services.

### Backup

Provide:

```text
scripts/backup_postgres.sh
```

The script reads `DATABASE_URL`, runs `pg_dump`, writes timestamped backups to a configurable backup directory, and supports a simple retention setting such as `BACKUP_KEEP_DAYS`.

### Tests And Quality

Database and API boundary tests run against PostgreSQL.

Each database test uses an isolated PostgreSQL schema.

Required contract coverage:

- `/api/me`
- UUID response formats
- settings per-user isolation
- model profiles per-user isolation without plaintext reveal
- encrypted secret storage and masked preview
- sessions CRUD and bundle
- cross-user resource access returns 404
- workflow ownership
- structured extraction ownership
- jobs queue, status, event stream, and permissions
- readiness checks

Pure logic tests do not need PostgreSQL.

Add `ruff` as a lightweight Python quality gate. Do not require `mypy` in Phase 1.

## Implementation Milestones

### M1: PostgreSQL Foundation

- Add SQLAlchemy engine and connection management.
- Add Alembic configuration and initial PostgreSQL migration.
- Add dedicated schema setup.
- Add UUID, jsonb, timestamptz conventions.
- Add development and production database configuration.

### M2: Users, Settings, Profiles, Secrets

- Add users and identity resolution.
- Implement `AUTH_MODE=dev-header`.
- Add `/api/me`.
- Migrate settings to per-user PostgreSQL storage.
- Migrate model profiles to per-user PostgreSQL storage.
- Add encrypted user secrets with Fernet.
- Remove plaintext reveal behavior.

### M3: Core Stores And User Boundary

- Migrate SessionStore, Job metadata reads, WorkflowStore, StructuredExtractionStore, artifact metadata, and related stores to PostgreSQL.
- Add ownership helpers.
- Add audit events.
- Add bundle endpoint.
- Keep literature corpus/index reads unchanged.

### M4: Queue And Worker

- Implement PostgreSQL-backed job claiming.
- Add worker process and heartbeat.
- Add JobHandlerRegistry.
- Migrate required background task types to worker.
- Preserve SSE progress using job_events.
- Add retry and cooperative cancellation.

### M5: Frontend User Scope

- Load `/api/me` first.
- Scope localStorage by internal user_id.
- Clear stale non-UUID active session entries.
- Switch session opening to bundle.
- Add development-only user switcher with full page reload.
- Display current user display name.

### M6: Operations, Tests, Docs

- Upgrade readiness.
- Update `dev.sh`.
- Add PostgreSQL-only local Docker Compose.
- Add backup script.
- Add production deployment documentation.
- Add PostgreSQL contract tests.
- Add ruff configuration.
- Mark SQLite platform-business code as legacy and out of runtime path.

## Open Follow-Up Items

These are intentionally not Phase 1 implementation requirements:

- Final login UI and account menu.
- JWT/OIDC/external auth implementation.
- Team or organization collaboration.
- PostgreSQL row-level security.
- Full user data export.
- User hard delete/compliance delete.
- Full containerized production deployment.
- Redis/Celery or distributed worker deployment.
- Object storage for artifacts.
- Migration of the literature corpus/index database.
